Суворо дотримуйся інструкцій нижче:

## 1. Чистота коду

1.1. Дотримуйся підходів Clean Architecture, MVVM, SOLID, DRY, KISS.
1.2. Пиши найменший можливий обсяг коду для виконання завдання.
1.3. Давай прості і зрозумілі назви, які дуже добре репрезентують, що робить той чи інший файл/змінна/функція/тощо навіть якщо вони довгі і написана в лоб. Наприклад, DeduplicationService.kt -> LocalEmbeddingsService.kt, extract() -> extractJsonFragment().
1.4. Не допускай `god object`. Максимум 300-400 рядків коду на файл.
1.5. Великі UI-блоки винось у окремі components/sections/files.

## 2. Хардкод

2.1. UI текст завжди винось у `app/src/main/res/values/strings.xml` і `app/src/main/res/values-en/strings.xml`.
2.2. Дати і час показуй через поточну локаль застосунку: `Locale.getDefault()`.
2.3. Числа і інші константні дані винось у const val CONSTANT_NAME.
2.4. Жоден текст не може бути захардкодений або написаний тільки з урахуванням однієї мови.

## 3. Help-Mode

3.1. Кожен ui елемент має підтримувати help-mode. Це режим, який відкривається через кнопку (?) справа зверху. При ньому основні блоки затемнюються і стають клікабельними. При натисненні з'являється пояснення.
3.2. Не накривай одну велику область, якщо секція складається з кількох окремих блоків: кожен логічний блок має відкриватися окремо.
3.3. Help-тексти завжди додавай у `values/strings.xml` і `values-en/strings.xml`. Формат: короткий маркерований список з `—`. Кожен пункт конкретний і чіткий.

## 4. Анімації

4.1. Не пиши локальні одноразові анімації, якщо вже є shared motion helpers.
4.2. Між головними екранами `Summary / Feed / Sources / Settings` анімацій не має бути взагалі. Для вкладених екранів, dialog, modal, … використовуй shared content/modal/visibility transitions.

## 5. Firebase Sync

5.1. Можливість синхронізації доступна для: джерел, підписок, збережених, налаштувань (крім API ключів), API-ключів.
5.2. Якщо нова модифікує щось із згаданого вище, то має пройти повний цикл синхронізації.
5.3. Потрібно оновити: локальне збереження, Firebase sync, mapper-и, DTO, backup/export/import, restore після входу на іншому пристрої, а також сумісність із уже існуючими даними.

## 6. Перевірка збірки

6.1. Після будь-яких модифікацій запусти збірку командою ./gradlew assembleDebug.
6.2. Якщо вона провалилася, але помилка банальна (імпорт, хвости, забув щось оновити), то виправ її без попередження.
6.3. Якщо проблема глибша, то відправ мені зрозуміле повідомлення, щоб я обрав подальший вектор руху.
